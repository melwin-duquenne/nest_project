Présentation du projet

Qu'est-ce que TaskFlow API ?

Une API REST complète de gestion de projets et de tâches, inspirée de Trello. Elle permet à des équipes de collaborer sur des projets, d'assigner des tâches et de recevoir des notifications en temps réel.

Ce que l'application permet

Gestion des utilisateurs
Inscription, connexion, rôles (admin, member, viewer)

Gestion des équipes
Création d'équipes, ajout / retrait de membres

Projets & tâches
Suivi du statut, priorités, assignation

Notifications temps réel
Alertes WebSocket à chaque assignation

Suppression en cascade

Supprimer une équipe supprime ses projets, qui suppriment leurs tâches. Si l'assigné d'une tâche est supprimé, la tâche survit (l'assigné passe à null).

Identifiants UUID

Tous les identifiants sont des UUID générés automatiquement — non-prédictibles, compatibles avec la distribution et les microservices.

Énumérations en base

Les statuts, priorités et rôles sont des ENUM PostgreSQL — l'intégrité est garantie au niveau de la base de données, pas seulement dans le code.

Base du projet

Ce qui était demandé
  • Créer et structurer un projet NestJS
  • Ajouter un endpoint de vérification de santé
  • Containeriser la base de données avec Docker
  • Organiser le code en modules distincts par domaine
  • Gérer les variables d'environnement de façon centralisée
Ce qui a été mis en place
  • Structure modulaire : utilisateurs, équipes, projets, tâches, commentaires
  • Base de données PostgreSQL lancée via Docker Compose
  • Variables d'environnement centralisées dans un fichier .env
  • Toutes les routes préfixées sous /api
  • Route /health pour vérifier que l'API répond correctement
Pourquoi

Poser une architecture modulaire dès le début évite d'accumuler de la dette technique. Chaque domaine métier dans son propre module rend le code lisible, testable et extensible sans impacter les autres parties de l'application.

CRUD & Validation

Ce qui était demandé
  • Implémenter toutes les opérations CRUD pour les utilisateurs
  • Gérer les erreurs : ressource introuvable et doublon d'email
  • Créer des objets de transfert de données pour les 5 entités
  • Valider automatiquement toutes les requêtes entrantes
  • Rejeter les champs non autorisés dans les requêtes
Ce qui a été mis en place
  • Contrôleur avec 5 routes REST (liste, détail, création, modification, suppression)
  • Service dédié à la logique métier, séparé du contrôleur
  • Règles de validation : email, énumération de rôle, longueur minimale
  • Rejet automatique avec message d'erreur clair pour toute donnée invalide
  • Héritage de structure entre DTOs pour éviter la duplication
Pourquoi

Séparer le contrôleur (routing HTTP) du service (logique métier) est fondamental en NestJS. Valider les données à l'entrée protège l'application : une requête invalide est rejetée immédiatement, avant d'atteindre la base de données.

Persistance avec TypeORM

Ce qui était demandé
  • Connecter l'application à PostgreSQL avec un ORM
  • Modéliser les 5 entités avec leurs propriétés
  • Gérer les relations entre entités (équipe–membres, projet–tâches)
  • Mettre en place un système de migrations de schéma
  • Peupler la base avec des données de test reproductibles
Ce qui a été mis en place
  • 5 entités TypeORM avec identifiants UUID et horodatage automatique
  • Mots de passe hachés avant stockage, jamais retournés dans les réponses
  • Relation Équipe–Utilisateurs via une table de jonction (many-to-many)
  • Système de migrations pour versionner les évolutions de schéma
  • Script de seed : 3 utilisateurs, 2 équipes, 2 projets, 3 tâches, 2 commentaires
Pourquoi

Remplacer le stockage en mémoire par une vraie base de données apporte la persistance des données. Les migrations permettent d'évoluer le schéma en production sans jamais perdre les données existantes, contrairement à la synchronisation automatique.

Authentification & Autorisation

Ce qui était demandé
  • Authentifier les utilisateurs avec des tokens JWT
  • Identifier l'utilisateur courant dans chaque requête
  • Restreindre certaines routes selon le rôle de l'utilisateur
  • Protéger toutes les routes par défaut
  • Empêcher un utilisateur de modifier le profil d'un autre
Ce qui a été mis en place
  • Vérification de l'email et du mot de passe à la connexion
  • Génération et validation de tokens JWT signés
  • Guard global : toutes les routes sont protégées par défaut
  • Décorateur pour marquer les routes publiques (login, etc.)
  • Rôles admin, member, viewer avec restrictions par route
  • Un utilisateur ne peut modifier que son propre profil
Pourquoi

Sécurité par défaut : une route oubliée reste protégée, jamais exposée accidentellement. Le système de rôles permet de gérer finement les permissions sans répéter la même logique dans chaque service.

Qualité & Documentation

Ce qui était demandé
  • Unifier le format des erreurs retournées par l'API
  • Gérer les erreurs de base de données (contrainte unique)
  • Logger chaque requête HTTP avec sa durée
  • Générer une documentation API interactive et navigable
  • Permettre de tester les routes directement depuis la doc
Ce qui a été mis en place
  • Filtre global qui formate toutes les erreurs de façon cohérente
  • Traduction des erreurs de contrainte unique en conflit 409
  • Journalisation de chaque requête : méthode, URL, statut, durée
  • Documentation Swagger accessible sur /docs
  • Tous les endpoints et champs annotés avec descriptions et exemples
Pourquoi

Une API sans logs est impossible à déboguer. Le filtre d'exceptions unifie les erreurs pour le front-end. Le Swagger devient le contrat vivant de l'API : les développeurs testent et comprennent les routes sans documentation externe à maintenir.

Tests automatisés

Ce qui était demandé
  • Tester le service utilisateurs en isolation
  • Tester le système de contrôle d'accès par rôles
  • Atteindre une couverture de code supérieure à 70%
  • Tester les flux d'authentification de bout en bout
  • Tester le CRUD complet avec une vraie base de données
Ce qui a été mis en place
  • Tests unitaires avec base de données simulée (mocks)
  • 4 scénarios pour le contrôle d'accès : libre, rôle correct, refusé, multi-rôles
  • Rapport de couverture de code généré à la demande
  • Tests de bout en bout avec une vraie base de données dédiée
  • Base de test réinitialisée et peuplée avant chaque test
Pourquoi

Les tests unitaires valident la logique en isolation, très rapidement, sans base de données. Les tests de bout en bout vérifient que toute la chaîne fonctionne ensemble. Les deux niveaux se complètent pour détecter les régressions avant la production.

Migration vers Prisma

Ce qui était demandé
  • Intégrer Prisma comme alternative à TypeORM
  • Définir tout le schéma en un seul fichier centralisé
  • Réécrire l'accès aux données pour les utilisateurs
  • Exclure les données sensibles des réponses
  • Explorer la base via l'interface graphique Prisma Studio
Ce qui a été mis en place
  • Schéma centralisé décrivant tous les modèles et les énumérations
  • Service Prisma partagé et disponible dans toute l'application
  • Opérations de lecture, création, modification et suppression réécrites
  • Mot de passe systématiquement exclu de toutes les réponses
  • Migrations générées et appliquées par ligne de commande
Pourquoi

Comparer deux ORMs en conditions réelles permet de comprendre leurs différences. Prisma offre un typage plus strict et un schéma unique, plus lisible que des annotations dispersées dans le code.

  • Authentifier les connexions WebSocket avec le même token JWT

Docker & CI/CD

Ce qui était demandé
  • Créer une image Docker optimisée pour la production
  • Configurer un environnement de production complet
  • Automatiser les tests et le build via GitHub Actions
  • Vérifier l'état de l'application depuis l'orchestrateur
  • Renforcer la sécurité des en-têtes HTTP
Ce qui a été mis en place
  • Build en deux étapes : construction puis exécution (image allégée sans dev)
  • Exécution sous un utilisateur non-root pour la sécurité
  • Endpoint de santé vérifié automatiquement par Docker
  • Docker Compose de production avec volumes de données persistants
  • Pipeline CI : vérification → tests unitaires → tests E2E → build Docker
Pourquoi

Le déploiement doit être reproductible et automatisé. Le build multi-étape produit une image légère. La pipeline CI détecte les régressions à chaque modification du code, avant qu'elles n'arrivent en production.
