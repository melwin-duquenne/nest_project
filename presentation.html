Cascade suppression

Team → Project → Task : CASCADE
Assignee supprimé : SET NULL — la tâche survit dans l'historique

UUID + ENUMs natifs PostgreSQL

IDs non-prédictibles. Statuts et rôles en ENUM — intégrité garantie au niveau base, pas seulement en code.

Démo — Auth JWT & Rôles

Authentification

  • POST /auth/login Login avec alice@example.com → retourne un access_token JWT
  • 401 Même route sans token sur /auth/me → 401 Unauthorized
  • 200 Même route avec le token → 200 + profil complet

Autorisation par rôles

  • 403 Login bob@example.com (member) → tenter POST /api/users → 403 Forbidden
  • 201 Login alice@example.com (admin) → même action → 201 Created

Choix techniques

Guards globaux

Toutes les routes sont protégées par défaut. On opte-in avec @Public(). Une route oubliée reste protégée, jamais exposée.

SET NULL vs CASCADE

Supprimer un user → assignee = NULL. Les tâches survivent. Supprimer un projet → CASCADE sur ses tâches.

TypeORM + Prisma

Deux ORMs pour comparer en conditions réelles. TypeORM gère migrations et CRUD. Prisma sur quelques lectures.

Difficultés rencontrées

Double ORM

Deux sources de vérité à maintenir en sync. Résolu en faisant TypeORM la source principale des migrations.

Auth WebSocket

HTTP upgrade ne supporte pas Authorization. Token lu depuis handshake.auth.token côté Socket.io.

Isolation tests e2e

dropSchema + seed dans beforeEach. Chaque test part d'un état prévisible et indépendant.
